# 변수 검증

## `{PLAN}` 검증

아래 규칙으로 `{PLAN}` 을 결정하고 검증해라:

- `{PLAN}` 이 비어있으면 다음 안내 문구를 출력하고 종료해라:
   ```
   ❌ 플랜 이름이 필요합니다. `/sbs-work {PLAN}` 형식으로 플랜 이름을 입력하세요.
   ```

- `.sisyphus/plans/{PLAN}.md` 파일이 존재하는지 확인해라. 존재하지 않으면 다음 안내 문구를 출력하고 종료해라.
   ```
   ❌ 플랜 파일이 존재하지 않습니다: .sisyphus/plans/{PLAN}.md
   ```

## `{TASK}` 검증

아래 규칙으로 `{TASK}` 을 결정하고 검증해라:

`{TASK}`: `.sisyphus/plans/{PLAN}.md` 에서 `- [ ]`(미완료 Task) 중 가장 첫 번째 Task

- `{TASK}`(미완료 태스크) 가 없으면 다음 안내 문구를 출력하고 종료해라.
   ```
   모든 태스크가 완료되었습니다.
   `/sbs-approve {PLAN}` 을 입력하여 작업을 종료해주세요.
   ```

## `{REQUEST_CHANGE}` 결정

- `{REQUEST_CHANGE}` 이 비어있으면 다음 안내 문구를 출력하고 종료해라.
   ```
   변경 요청 내용을 입력해주세요: `/sbs-reject {PLAN} {변경 요청 내용}`
   ```

# 반려 작업 시작

## **최상단에 반드시 아래 정보를 출력해라:**
```
- 플랜: {PLAN}
- 태스크: {TASK}
- 요청 내용 : {REQUEST_CHANGE}
**`{TASK}` 에 대한 수정 요청 사항을 반영하고, 다시 검수 요청을 드리겠습니다.**
```

## 반려 작업 프로세스:

`.sisyphus/plans/{PLAN}.md`에서 가장 최근 검수 요청된 미완료 Task(`- [ ]`)를 찾아라.

필수 규칙:
- 코드 변경을 최우선으로 수행하여 `<변경 요청 내용>`을 해당 Task 구현에 즉시 반영
- 변경 요청 영향 범위가 현재 Task를 넘어가면 플랜 파일의 관련 Task 설명/[GATE]를 필요한 만큼만 수정
- 영향이 없는 Task는 수정 금지
- 수정 후 즉시 멈추고 다음 Task 진행 금지
- 해당 Task의 `[GATE]` 기준으로 재검수 요청
- 사용자 approve 전 체크박스를 `- [x]`로 변경 금지 (`- [ ]` 유지)

재검수 요청에 반드시 포함:
- 수행한 Task 번호/제목
- 변경 요청 내용 요약
- 변경 파일 목록
- 변경 요약 (이전 대비 달라진 점)
- 실행한 검증 결과(가능하면 lint/typecheck/test 중 1개 이상)
- `[GATE]` 검수 포인트/성공 기준 체크리스트

## **마지막에 반드시 아래 정보를 출력해라:**
```
{검수 요청 내용}

작업을 완료하였습니다.
사용자의 다음 입력을 대기하겠습니다:
- 승인: `/sbs-approve {PLAN}`
- 반려(변경 요청): `/sbs-reject {PLAN} "{변경 요청 내용}"`

### PLAN APPROVE QUESTION
{ "PLAN": "{PLAN}", "TASK": "{TASK}" }
```
